SaMD（Software as Medical Device）とは

SaMDの定義

SaMDは「医療機器としてのソフトウェア」を指します。従来のハードウェア医療機器とは異なり、 ソフトウェア単体で医療目的を果たす製品です。診断支援、治療計画、患者モニタリングなど、 医療現場で直接使用されるアプリケーションが該当します。

診断支援

AI画像診断など

モニタリング

バイタルサイン監視

治療計画

投薬量計算など

SaMD開発の特殊性

  • 厳格な規制要件（薬機法、FDA等）
  • 高度なセキュリティ要件
  • 品質管理システム（QMS）の必要性
  • 長期的な保守・監査証跡

フロントエンド・バックエンドの理解

フロントエンド (Frontend)

ユーザーが直接操作する部分。ブラウザで動作するWebアプリケーションです。 SaMDでは特に使いやすさと正確性が重要になります。

主要技術

  • Vite: 高速なビルドツール
  • React/Vue: UIフレームワーク
  • TypeScript: 型安全性
  • Tailwind CSS: スタイリング

// 開発サーバー起動

npm run dev

// 本番用ビルド

npm run build

バックエンド (Backend)

データ処理やビジネスロジックを担当する部分。 SaMDでは患者データの安全な管理と処理が最重要です。

  • API: データ送受信
  • 認証: アクセス制御
  • ビジネスロジック: 医療計算処理

// Supabase関数デプロイ

supabase functions deploy

// ローカル環境起動

supabase start

Docker技術群の完全理解

Docker

アプリケーションをコンテナという軽量な仮想環境で実行する技術。 「どの環境でも同じように動く」を実現します。

利点

  • • 環境の統一性
  • • 高速な起動
  • • リソース効率
  • • 移植性の高さ

Dockerfile

Dockerイメージの作成手順を記述したテキストファイル。 「レシピ」のような役割を持ちます。

# Node.js環境を使用

FROM node:18-alpine

# 作業ディレクトリ設定

WORKDIR /app

# 依存関係をコピー

COPY package*.json ./

# パッケージインストール

RUN npm install

# アプリケーションコードをコピー

COPY . .

# アプリケーション起動

CMD ["npm", "start"]

Dockerイメージ

アプリケーションとその実行環境を含む「テンプレート」。 読み取り専用で、コンテナの基となります。

特徴

  • • 不変性（Immutable）
  • • レイヤー構造
  • • 共有可能
  • • バージョン管理

Dockerコンテナ

Dockerイメージから作成される実行中のプロセス。 アプリケーションが実際に動作する環境です。

特徴

  • • 軽量で高速起動
  • • 分離された環境
  • • 動的な状態
  • • スケーラブル

Docker Compose

複数のコンテナを組み合わせたアプリケーションを定義・実行するツール。 YAMLファイルで構成を記述します。

# docker-compose.yml

version: '3.8'

services:

app:

build: .

ports:

- "3000:3000"

db:

image: postgres:15

environment:

POSTGRES_PASSWORD: secret

利点

  • • 複数サービスの一括管理
  • • ネットワーク自動設定
  • • 開発環境の統一
  • • 簡単なスケーリング

OCIイメージ

Open Container Initiative準拠のコンテナイメージ標準。 Docker以外のランタイムでも使用可能です。

特徴

  • • 標準化された仕様
  • • 互換性の確保
  • • エコシステムの拡張

Container Registry

Dockerイメージを保存・配布するためのサービス。 Docker Hub、AWS ECR、GitHub Container Registryなど。

機能

  • • イメージの保存・管理
  • • バージョン管理
  • • アクセス制御
  • • 自動ビルド

Supabaseエコシステム

Supabase概要

オープンソースのFirebase代替。PostgreSQLをベースとしたBaaS（Backend as a Service）プラットフォーム。 SaMD開発に必要な機能を包括的に提供します。

Database

Auth

Storage

Functions

Supabase CLI

Supabaseプロジェクトの管理・開発を行うコマンドラインツール。 ローカル開発環境の構築からデプロイまで対応。

# プロジェクト初期化

supabase init

# ローカル環境起動

supabase start

# マイグレーション実行

supabase db push

# 関数デプロイ

supabase functions deploy

主要機能

  • • プロジェクト管理
  • • データベース操作
  • • 関数の開発・デプロイ
  • • ローカル開発環境

Supabase Edge Functions

Deno上で動作するサーバーレス関数。 カスタムAPIエンドポイントやビジネスロジックの実装に使用。

// functions/hello/index.ts

import { serve } from "std/http/server.ts"

serve(async (req) => {

return new Response(

JSON.stringify({ message: "Hello!" }),

{ headers: { "Content-Type": "application/json" } }

)

})

特徴

  • • TypeScript/JavaScript対応
  • • グローバル配信
  • • 自動スケーリング
  • • 低レイテンシ

Postgres (Supabase DB)

Supabaseのコアとなるデータベース。PostgreSQLの強力な機能をフルに活用できます。 SaMDに必要なデータ整合性・セキュリティ要件を満たします。

セキュリティ

  • • Row Level Security (RLS)
  • • 暗号化機能
  • • 監査ログ
  • • アクセス制御

機能

  • • JSON/JSONB型
  • • 全文検索
  • • 拡張機能
  • • 関数・トリガー

運用

  • • 自動バックアップ
  • • 高可用性
  • • 監視機能
  • • スケーリング

ビルド・デプロイメントの理解

ビルド (Build)

ソースコードを本番環境で実行可能な形式に変換するプロセス。 コンパイル、最適化、バンドルなどを含みます。

ビルドプロセス

  • • TypeScript → JavaScript変換
  • • モジュールバンドル
  • • コード最適化・圧縮
  • • 静的アセット処理

# 開発用ビルド（高速）

npm run dev

# 本番用ビルド（最適化）

npm run build

デプロイ (Deploy)

ビルドされたアプリケーションを本番環境に配置し、 ユーザーがアクセスできる状態にするプロセス。

デプロイ対象

  • • 静的ファイル（HTML/CSS/JS）
  • • サーバーレス関数
  • • データベースマイグレーション
  • • 環境変数設定

# Vercelデプロイ

vercel --prod

# Supabase関数デプロイ

supabase functions deploy

Vite: 次世代ビルドツール

ES modulesを活用した高速なビルドツール。開発時の高速なHMR（Hot Module Replacement）と 本番時の最適化されたビルドを提供します。

開発時の利点

  • • 高速な起動時間
  • • インスタントHMR
  • • TypeScript対応
  • • 豊富なプラグイン

本番時の利点

  • • 効率的なバンドル
  • • Code Splitting
  • • Tree Shaking
  • • 最適化された出力

Static Hosting / CDN

静的ファイルをグローバルに配信するサービス。 Vercelでは自動的にCDNが設定されます。

特徴

  • • 世界中のエッジサーバー
  • • 高速なコンテンツ配信
  • • 自動キャッシュ最適化
  • • SSL/TLS暗号化

APIサーバー

データの処理やビジネスロジックを提供するサーバー。 Supabaseでは自動的にREST APIが生成されます。

機能

  • • 自動生成REST API
  • • リアルタイム購読
  • • 認証・認可
  • • カスタムエンドポイント

CI/CD・環境管理・デプロイメントフロー

CI/CD (Continuous Integration/Continuous Deployment)

コードの統合・テスト・デプロイを自動化するプロセス。 SaMD開発では品質保証と規制要件対応のために特に重要です。

CI (継続的インテグレーション)

  • • 自動テスト実行
  • • コード品質チェック
  • • セキュリティスキャン
  • • ビルド確認

CD (継続的デプロイメント)

  • • 自動デプロイ
  • • 環境間の移行
  • • ロールバック機能
  • • デプロイメント監視

Staging環境

本番環境と同じ構成でテストを行う環境。 SaMDでは薬事申請前の最終検証に使用します。

用途

  • • 統合テスト
  • • ユーザー受け入れテスト
  • • パフォーマンステスト
  • • 最終品質確認

Production本番環境

実際に医療従事者が使用する環境。 最高レベルの可用性とセキュリティが要求されます。

要件

  • • 高可用性（99.9%以上）
  • • 24/7監視
  • • 定期バックアップ
  • • 規制遵守
